branching

creating branches

we can create another 'branch' of the codebase with the checkout command

    
      $ git checkout -b part-two

we can push this branch to github with git push

      
        $ git push origin part-two

let's create a bunch of branches and play around

branching

cherry-picking

cherry picking allows us to grab a commit from a different branch, even if it's out of order

this is helpful for keeping bug-fixes aligned across two separate versions of software

    
      $ git checkout master
      .. EDIT JS
      $ git add .
      $ git commit -m 'edited js'
      [master e466ea1] edited js
       1 files changed, 2 insertions(+), 0 deletions(-)
      $ git checkout part-two
      Switched to branch 'part-two'
      $ git cherry-pick e466ea1
      [part-two 481d3d3] edited js
       1 files changed, 2 insertions(+), 0 deletions(-)

merging branches

eventually we want to merge our branches together

    
      $ git merge BRANCH

of course, there is always the possibility of a merge conflict here

merging remotes

when we're working with a team, we want to merge in the team member's changes

    
      $ git fetch

      $ git branch -a
        master
      * part-two
        remotes/origin/master
        remotes/origin/part-two

of course, there is always the possibility of a merge conflict here

fixing mistakes

  • reset
    — for uncommitted, but staged, files 
            
          $ git reset --hard HEAD^
          $ git reset --soft HEAD~3
  • revert
    — for committed files 
            
          $ git revert HEAD^
          $ git revert HEAD~3

    creates a new commit in front of the mistake commit
